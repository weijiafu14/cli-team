# 屏幕素材监视 Agent 调研与建设方案 (V1.0)

## 1. 调研背景与问题陈述

自媒体创作者（特别是 “Build in Public” 倡导者）在工作过程中常需截取屏幕素材以展示进展。目前手动截图效率低，且容易中断心流。

**核心需求：**

- **目标导向**：根据设定的工作目标（如“设计 AionUi 官网”）自动开启。
- **静默采集**：后台自动截图，不干扰用户。
- **智能筛选**：自动过滤重复画面，整理有价值的素材。
- **AI 辅助输出**：工作结束时自动生成社交媒体总结草稿。

## 2. 市场竞品调研 (2026)

| 产品                 | 优势                     | 不足                                                    |
| :------------------- | :----------------------- | :------------------------------------------------------ |
| **Screenpipe**       | 开源、高性能、事件驱动。 | 仅提供原始数据流，缺乏目标驱动的逻辑和 Agent 协同能力。 |
| **Microsoft Recall** | 系统级集成，搜索能力强。 | 隐私争议大，主要面向“记忆搜索”而非“素材策展”。          |
| **Rewind.ai**        | 记录一切。               | 闭源商业软件，难以与自定义 AI Agent 工作流深度集成。    |

**调研结论：** 市场上尚无完美结合“目标驱动 + Agent 自动总结”的轻量化产品。**基于 AionUi 自研**具有天然优势（可复用 Cron 系统、主进程权限、Agent 协作链路）。

## 3. 推荐架构设计

### 3.1 核心组件 (Main Process Service)

- **ScreenCaptureSessionService**: 驻留在 Electron 主进程的服务，负责实际的截图调度。
- **Deduplication Logic (dHash)**: 使用感知哈希算法对比相邻帧，当 Hamming 距离小于阈值时自动跳过，节省 80%+ 的无效存储。
- **Session Manifest**: 每个会话生成一个 `manifest.json`，记录目标、时间、关键帧哈希。

### 3.2 交互指令 (Command Protocol)

Agent 通过输出特定指令与系统交互，无需用户感知：

- `[SCREEN_MONITOR_START: goal="...", interval=5]`
- `[SCREEN_MONITOR_STOP]`
- `[SCREEN_MONITOR_STATUS]`

### 3.3 数据流向

1. **用户**：告诉 Agent “我要开始工作了，目标是 X”。
2. **Agent**：输出启动指令，激活主进程服务。
3. **主进程**：定时（如每 5 分钟）截图 -> dHash 比对 -> 存入 `uploads/monitoring/[sessionId]/`。
4. **Agent**：在用户说“结束”时停止会话，并调用 Vision 模型分析保存的关键帧，生成总结。

## 4. 与 Claude "Computer Use" 的对比分析

针对用户提出的 Claude 3.5 Sonnet "Computer Use" 功能，本方案进行了深度对比：

| 维度           | Claude Computer Use (主动型)                                 | AionUi 素材监视方案 (被动型)                                                  |
| :------------- | :----------------------------------------------------------- | :---------------------------------------------------------------------------- |
| **核心逻辑**   | **它用电脑**：模型看屏幕并操作鼠标键盘，代替人执行任务。     | **它看你用电脑**：在后台静默记录人的操作过程。                                |
| **应用场景**   | **自动化任务**：如“帮我订票”、“去这个网站填表”。             | **素材生产**：如“记录我开发这个仓库的过程”。                                  |
| **成本/Token** | **极高**：每一步操作都需要上传高分辨率截图，Token 消耗极快。 | **极低**：本地静默截图+dHash去重，仅在总结时消耗少量 Token。                  |
| **真实性**     | **AI 过程**：展示的是 AI 工作的能力。                        | **人的过程**：展示的是人类创作者的真实思考和产出，符合“Build in Public”灵魂。 |
| **适用性**     | 适合无人值守的自动化任务。                                   | 适合创作者、开发者等需要展示个人工作过程的自媒体素材采集。                    |

**结论：** 虽然 Claude 的 "Computer Use" 很强大，但它的定位是“执行者”而非“观察者”。对于自媒体素材库而言，我们需要的是一个**不干扰心流、低成本、高效率**的“数字随行摄影师”，而这正是本方案的差异化价值所在。

## 5. 架构优化与风险控制 (根据评审反馈)

1. **权限提示**：首次启动前需引导用户开启操作系统“屏幕录制”权限。
2. **敏感应用过滤**：未来可支持应用黑名单（如 1Password 等），截图时自动避开。
3. **存储自动清理**：支持设定素材保留期限，过期自动清理以防磁盘占满。
