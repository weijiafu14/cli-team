# 调研设计说明：AionUi 基座之上的定制改造

## Problem Statement

用户要求基于仓库内文档和 git 提交，梳理这个分支相对上游 AionUi 主干到底做了什么，而不是只看某一个文件。

## Chosen Approach

以 `origin/main` 作为 AionUi 基线，结合 `git log origin/main..HEAD`、`git diff --stat origin/main..HEAD`、README 和 `docs/tech/agent-team/*` / `docs/development-notes.md` 交叉验证，按功能层次归纳改动。

## Alternatives Considered

- 只看 README：信息足够概括，但缺少代码和提交级证据。
- 只看 diff：能看到实现面，但不容易快速理解产品意图和演进顺序。
- 最终采用文档 + 提交 + 关键代码三方交叉验证。

## Affected Files And Interfaces

- 后端/主进程：`src/process/services/agentTeam/*`、`src/process/bridge/agentTeamBridge.ts`、`src/common/ipcBridge.ts`
- 数据模型：`src/common/storage.ts`、`src/process/database/{schema,migrations,types}.ts`
- 任务路由：`src/process/bridge/conversationBridge.ts`、`src/process/task/*`、`src/process/services/cron/WorkerTaskManagerJobExecutor.ts`
- 前端：`src/renderer/pages/guid/components/TeamBuilder.tsx`、`src/renderer/pages/conversation/platforms/agent-team/*`、`src/renderer/pages/conversation/GroupedHistory/*`
- Codex 续连：`src/agent/acp/acpConnectors.ts`、`src/agent/codex/core/CodexAgent.ts`、`src/process/task/CodexAgentManager.ts`
- 文档：`readme.md`、`docs/tech/agent-team/*`、`docs/development-notes.md`

## Risks And Follow-up Checks

- 当前工作树存在用户未提交改动，调研结论应只以 `origin/main..HEAD` 已提交历史为准。
- `package.json` 有若干依赖版本回退/新增，若后续要解释其必要性，还需要额外核对运行或兼容性背景。
- 若要进一步回答“哪些是最小可合并集合”，还需要把 snapshot 类提交从逻辑改动中剥离。

## Conclusion

相对 AionUi 主干，这个分支的核心不是修补单点，而是引入了一套名为 Agent Team 的多智能体协作能力：

1. 新增 `agent-team` 会话类型，让多个真实子 agent 共用同一 workspace，由父会话做编排壳层。
2. 在 workspace 内落地 `.agents/teams/<teamId>/coord/` 文件协议，提供消息流、锁、附件、状态游标和调度脚本。
3. 新增主进程调度器与文件监听器，把协调流转换成对 Claude/Codex/Gemini 子会话的唤醒、转发和共识约束。
4. 新增前端 TeamBuilder、AgentTeamChat、侧边栏嵌套团队展示，形成可创建、可观察、可投递文件的团队交互界面。
5. 修复多团队隔离、文件附件持久化、时间排序、共识 ACK 校验等协议级问题。
6. 为 Codex 增加原生 session ID 持久化与本地 `codex-acp v0.10.0` 优先连接，解决跨进程 resume 稳定性。
7. 补齐 README、架构文档、UI 文档、协议文档和开发 notes，使这套 fork 特性有完整说明。
